# PathLogs

Система управления проектами с **ветвлением задач** (как в git), **канбан-доской** (как в Trello) и **патч-логами** — полной историей реализации каждой задачи.

## Стек

| Слой | Технология |
|---|---|
| Фреймворк | Next.js 16 (App Router, FullStack, TypeScript) |
| БД | PostgreSQL 16 в Docker + Prisma ORM |
| Авторизация | Auth.js (NextAuth v5), JWT-сессии, bcrypt, роли, Google OAuth |
| Визуализация веток | React Flow (@xyflow/react) |
| Стили | TailwindCSS 4, тёмная тема |
| Файлы | S3-совместимое хранилище (S3 / R2 / MinIO) с автоматическим fallback на локальный диск |

## Запуск

```bash
# 1. Поднять PostgreSQL
docker compose up -d postgres

# 2. Установить зависимости и применить миграции
npm install
npx prisma migrate dev

# 3. (опционально) Демо-данные
node prisma/seed.mjs

# 4. Запустить
npm run dev
```

**Тесты** (права доступа и server actions; используют отдельную БД `pathlogs_test` в том же Postgres):

```bash
npm test
```

Открыть http://localhost:3000

**Демо-аккаунты** (после seed, пароль у всех `demo1234`):

- `admin@pathlogs.dev` — администратор
- `manager@pathlogs.dev` — менеджер
- `analyst@pathlogs.dev` — аналитик
- `dev@pathlogs.dev` — разработчик

Без seed: первый зарегистрированный пользователь автоматически становится администратором.

## Возможности

- **Проекты** с ключами (как в Jira: `PAY-12`), участниками, прогрессом и архивированием
- **Задачи с ветвлением**: у любой задачи могут быть подзадачи любой глубины — дерево отображается на интерактивном **графе веток** (цвет ребра = статус, анимация = «в работе», пунктир = связи «блокирует / связана / дублирует»)
- **Канбан-доска** с drag & drop по колонкам: К выполнению → В работе → На проверке → Готово (+ статусы Закрыта и В архиве)
- **Кастомные колонки** доски: добавляйте свои (например, «Тестирование») — перенос в них не меняет статус задачи; стандартные колонки привязаны к статусам. Удаление кастомной колонки возвращает задачи в колонки их статусов
- **WIP-лимиты** колонок: клик по счётчику задаёт лимит; при превышении колонка подсвечивается красным
- **Шаблоны задач** (например «Релиз»): тип, приоритет, описание и готовый чек-лист — выбираются в форме создания и предзаполняют поля
- **Перекраска** карточек задач и колонок — палитра по клику на цветную точку колонки или кружок на карточке
- **Чек-листы** в задачах: прогресс-бар, инлайн-редактирование, начальный чек-лист прямо в форме создания («пункт на строку»)
- **Markdown (ограниченный)** в описании задач, патч-логах и пунктах чек-листа: **жирный**, *курсив*, `код`, блоки кода, списки, цитаты, безопасные ссылки. Рендер в React-элементы — raw-HTML и XSS исключены, размеры заголовков ограничены, чтобы не ломать вёрстку
- **Патч-лог** у каждой задачи — журнал реализации с автором и временем
- **Обсуждение**: комментарии к задачам с Markdown и **@упоминаниями** участников (автодополнение по `@`, упомянутый получает персональное уведомление, имя подсвечивается)
- **Уведомления**: «вас назначили», «новый комментарий», «запись в патч-логе», «статус изменился» — счётчик в сайдбаре, страница `/notifications`, просмотр помечает прочитанным
- **Мои задачи** (`/my`): все открытые задачи пользователя по всем проектам — группировка по статусу, сортировка по сроку, просроченные подсвечены
- **Диаграмма Ганта** (вкладка): задачи с датами как полосы на дневной шкале, цвет по статусу, линия «сегодня», подсветка просроченных; **полосу можно перетаскивать** (сдвиг дат) и **тянуть за края** (начало/срок) — изменения сразу сохраняются
- **Аналитика проекта** (вкладка): распределение по статусам/типам/приоритетам, **динамика по неделям** (создано vs закрыто), «оценка vs факт» по задачам, часы и **стоимость** по сотрудникам
- **Учёт ставок и стоимости**: админ задаёт ставку/час пользователю — стоимость трудозатрат считается в аналитике и попадает в XLSX
- **Сохранённые фильтры** списка задач (на пользователя и проект): сохраняйте текущий набор фильтров и применяйте в один клик
- **Горячие клавиши**: `g d` — проекты, `g m` — мои задачи, `g n` — уведомления, `g p` — профиль, `?` — подсказка
- **Профиль** (`/profile`): статистика, смена имени и пароля; OAuth-аккаунты могут установить пароль для входа по email
- **Трудозатраты**: оценка в часах, записи времени с комментариями, прогресс-бар «потрачено / оценка»
- **Категоризация**: тип (фича, баг, аналитика, менеджмент, дизайн…), приоритет, даты начала и срока
- **Несколько исполнителей** на задачу, связи между задачами
- **Файлы** к задачам: загрузка в S3/R2, при недоступности — локально в `./uploads`
- **Админка**: пользователи, роли (админ / менеджер / аналитик / разработчик), деактивация, статистика
- **Права доступа**: проекты видны только участникам (админ видит всё); управление участниками, колонками и архивом — у владельца проекта, менеджеров и админа; удаление задачи — у её автора, менеджера или владельца; файлы задач доступны только участникам проекта
- **Список задач** с поиском и фильтрами по статусу, типу, исполнителю
- **Экспорт**: проект в **XLSX** (листы «Задачи» и «Трудозатраты», exceljs) и печатный **PDF-отчёт** (светлая вёрстка, сводка, задачи по статусам, часы по сотрудникам — сохраняется через диалог печати браузера)
- **Календарь**: задача с датами добавляется в **Google Calendar** одной кнопкой (без API-ключей) или скачивается как **.ics** (Outlook / Apple Calendar); весь проект выгружается одним `.ics`-файлом со всеми задачами, у которых есть даты
- **Вход через Google (OAuth)** — опционально: задайте `AUTH_GOOGLE_ID` и `AUTH_GOOGLE_SECRET` в `.env`, кнопка появится на странице входа. Пользователь создаётся автоматически (первый в системе — админ), деактивированные не допускаются
- **API-токены** (`/profile`): персональные токены для интеграций и git-вебхука; в БД хранится только SHA-256-хэш, сам токен показывается один раз
- **Интеграции / исходящие webhooks** (на проект): Slack (Incoming Webhook), Telegram (Bot API) или произвольный JSON-приёмник — события задач уходят в подключённые каналы
- **Git-вебхук**: упоминание `PAY-12` в сообщении коммита автоматически создаёт запись в патч-логе задачи (защита от дублей по паре задача+sha) — см. ниже
- **Еженедельный отчёт на почту**: cron-эндпоинт рассылает участникам сводку по проекту (создано/закрыто/просрочено/часы); без SMTP — пишет в консоль

## Интеграции через API

Создайте токен в **Профиле** → API-токены и передавайте его в заголовке `Authorization: Bearer <токен>`.

**Git-вебхук** — `POST /api/git/webhook` (тело совместимо с push-хуком GitHub/GitLab):

```json
{ "commits": [{ "id": "<sha>", "message": "PAY-12 починил оплату", "url": "https://…" }] }
```

Каждое упоминание `КЛЮЧ-НОМЕР` превращается в запись патч-лога соответствующей задачи. Токен должен принадлежать участнику её проекта.

**Еженедельный отчёт** — `POST /api/cron/weekly-report` (токен администратора). Поставьте на cron (например, по понедельникам). `?projectId=…` — отчёт только по одному проекту. Для реальной отправки задайте `SMTP_*` в `.env`.

## Google OAuth

1. В [Google Cloud Console](https://console.cloud.google.com/apis/credentials) создайте OAuth Client ID (тип Web).
2. Добавьте redirect URI: `http://localhost:3000/api/auth/callback/google` (и продакшен-адрес).
3. Пропишите в `.env`: `AUTH_GOOGLE_ID=…` и `AUTH_GOOGLE_SECRET=…` — кнопка «Войти через Google» появится сама.

## Хранилище файлов

Заполните в `.env` переменные `S3_*` (работает с AWS S3, Cloudflare R2, MinIO). Если они пусты или загрузка в S3 падает — файл автоматически сохраняется локально в `./uploads` и раздаётся через `/api/files/[key]` (только авторизованным).
